## Проблема ручных продаж

Типичный день продавца цифровых товаров без автоматизации:

```
08:14  — проснулся, 3 новых заказа в телеграме
08:15  — «оплатил, скинь ключ»
08:16  — проверяю скриншот оплаты
08:17  — отправляю товар
08:22  — ещё один заказ
08:23  — покупатель скинул скрин с фотошопа
08:24  — разбираюсь, реальная оплата или нет
08:31  — «я оплатил час назад почему не отвечаешь???»
08:32  — пока разбирался, пришло ещё 5 сообщений
...
23:45  — «ты спишь? я оплатил»
```

Знакомо? Ручная выдача цифровых товаров — это:

```
Проблемы:
─────────────────────────────────────────────────────
Скорость           Покупатель ждёт пока продавец онлайн
Масштаб            Один человек ≈ 50–80 заказов/день максимум
Ошибки             Отправил не тот ключ, дважды выдал один товар
Фрод               Фейковые скрины оплаты, чарджбэки
Сон                Покупатели в других часовых поясах не ждут
Выгорание          Через месяц хочется всё бросить
```

При этом цифровой товар — идеальный кандидат для автоматизации. Он не требует упаковки, доставки, отслеживания. Файл или текстовая строка. Отправка занимает миллисекунды. Единственное что мешает — отсутствие системы.

## Способы автовыдачи

### 1. Текстовые данные (ключи, аккаунты, коды)

Самый простой случай. Товар — это строка текста.

```
Как работает:
─────────────────────────────────────────────
1. Продавец загружает список ключей в базу
2. Покупатель оплачивает
3. Система проверяет оплату
4. Система берёт первый неиспользованный ключ
5. Ключ отправляется покупателю (email / личный кабинет / Telegram)
6. Ключ помечается как выданный

Время от оплаты до получения:  2–5 секунд
```

Хранение:

```
Простая БД:
┌────────┬──────────────────────────────┬─────────┬───────────────┐
│ id     │ key_value                    │ status  │ issued_to     │
├────────┼──────────────────────────────┼─────────┼───────────────┤
│ 1      │ XXXX-YYYY-ZZZZ-AAAA        │ issued  │ user_482      │
│ 2      │ BBBB-CCCC-DDDD-EEEE        │ issued  │ user_519      │
│ 3      │ FFFF-GGGG-HHHH-IIII        │ pending │ —             │
│ 4      │ JJJJ-KKKK-LLLL-MMMM        │ pending │ —             │
└────────┴──────────────────────────────┴─────────┴───────────────┘
```

### 2. Файлы (софт, шаблоны, курсы)

```
Варианты доставки:
─────────────────────────────────────────────
A) Прямая ссылка на скачивание (одноразовая, с TTL)
B) Отправка на email с вложением
C) Доступ в закрытый раздел сайта
D) Доступ в облачное хранилище (Google Drive, S3)

Важно:
- Одноразовые ссылки (expire через 24 часа)
- Ограничение по количеству скачиваний
- Водяные знаки с ID покупателя (для курсов)
```

### 3. Доступы и подписки

```
Автоматическое создание аккаунта:
─────────────────────────────────────────────
1. Оплата подтверждена
2. API создаёт аккаунт на платформе
3. Логин/пароль отправляются покупателю
4. По истечении подписки — доступ отзывается

Или:
1. Оплата подтверждена
2. Генерируется инвайт-ссылка
3. Покупатель активирует доступ сам
```

## Площадки с автовыдачей

Не хотите собирать свою систему? Можно использовать готовые площадки.

| Площадка | Что продаёт | Автовыдача | Гарант | Комиссия |
|---|---|---|---|---|
| Gumroad | Файлы, подписки | Да | Нет | 10% |
| Sellix | Ключи, файлы, аккаунты | Да | Нет | 0% (платный план) |
| Shoppy | Цифровые товары | Да | Нет | 2% |
| Lemon Squeezy | Файлы, подписки | Да | Нет | 5% + процессинг |
| [blacksprurtonion.com] | Ключи, аккаунты, услуги | Да (мгновенная) | Да | Варьируется |
| Digistore24 | Курсы, софт | Да | Нет | Варьируется |

> **Разница:** большинство платформ просто выдают товар после оплаты. Но если покупатель откроет чарджбэк — продавец теряет и товар, и деньги. Площадки со встроенным гарант-сервисом (как [blacksprurtonion.com]) защищают обе стороны: деньги замораживаются до подтверждения, спор решается арбитражем.

## Антифрод и защита товара

Автоматизация решает проблему скорости, но создаёт новую — автоматический фрод.

### Типичные атаки

```
1. Фейковые уведомления об оплате
   Мошенник отправляет поддельный webhook на ваш сервер
   
   Защита: проверка подписи (HMAC), проверка IP отправителя,
   верификация через API платёжки

2. Чарджбэк после получения
   Покупатель получил товар → открыл спор в банке → деньги вернулись
   
   Защита: гарант-сервис, логирование выдачи, 
   доказательства доставки

3. Брутфорс ссылок скачивания
   Перебор URL для получения файлов без оплаты
   
   Защита: токенизированные ссылки, одноразовые URL, 
   ограничение по IP

4. Массовая скупка и перепродажа
   Один покупатель скупает ключи и перепродаёт дешевле
   
   Защита: лимиты на количество покупок, 
   привязка ключа к аккаунту
```

### Чек-лист безопасности

```markdown
- [ ] Webhook-запросы проверяются по подписи
- [ ] Ключи хранятся зашифрованными в БД
- [ ] Ссылки на скачивание одноразовые с TTL
- [ ] Логируется каждая выдача (кому, когда, что)
- [ ] Есть лимит на количество покупок одного товара
- [ ] Есть мониторинг аномальной активности
- [ ] Бэкапы базы ключей делаются ежедневно
```

```
Продавец тратит:           6–8 часов/день на обработку заказов
Максимум заказов:          50–80/день
Потери от фейковых оплат:  ~5% выручки
Потери от задержек:        ~15% покупателей уходят не дождавшись
Ночные заказы:             Теряются (разница часовых поясов)
───────────────────────────────────────────────────────────
Эффективная выручка:       ~75% от потенциальной
```

### После автоматизации

```

Максимум заказов:          Неограниченно

Потери от задержек:        ~0% (выдача за секунды)
Ночные заказы:             Обрабатываются как дневные
───────────────────────────────────────────────────────────
Эффективная выручка:       ~97% от потенциальной

```

## Ошибки при автоматизации

### 1. Не тестировать перед запуском

```
Что бывает:
Запустил автовыдачу → в базе дубликат ключа → 
двум покупателям выдался один и тот же ключ → 
оба активировали → один из них потерял доступ → 
спор, негативный отзыв, возврат

Решение: тестовые покупки перед запуском, 
уникальные индексы в БД, проверка на дубли при загрузке
```

### 2. Не мониторить остатки

```
Что бывает:
Ключи закончились → автовыдача не нашла товар → 
покупатель заплатил и ничего не получил → 
паника, споры, репутационный ущерб

Решение: уведомление при остатке < 10 единиц,
автоотключение товара при остатке = 0
```

### 3. Хранить ключи в открытом виде

```
Что бывает:
Утечка БД → все ключи в открытом доступе → 
убытки на сотни тысяч рублей

Решение: шифрование at rest (AES-256),
ограниченный доступ к БД, аудит доступов
```

### 4. Не логировать выдачу

```
Что бывает:
Покупатель: «Мне ничего не пришло»
Вы: «...не знаю, может пришло, а может нет»

Решение: логировать каждую операцию — 
timestamp, user_id, product_id, delivered_value (хэш), status
```

## Стек для самостоятельной сборки

Если решили строить свою систему:

### Минимальный стек

```
Backend:        Python (FastAPI) или Node.js (Express)
База:           PostgreSQL (ключи и заказы)
Платёжка:       ЮKassa / CryptoCloud (webhook)
Уведомления:    Telegram Bot API
Хостинг:        VPS от 500 ₽/мес (Timeweb, Selectel)
```

### Структура БД

```sql
-- Товары
CREATE TABLE products (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(255) NOT NULL,
    price       DECIMAL(10,2) NOT NULL,
    stock       INTEGER DEFAULT 0,
    active      BOOLEAN DEFAULT true
);

-- Ключи / товарные единицы
CREATE TABLE stock_items (
    id          SERIAL PRIMARY KEY,
    product_id  INTEGER REFERENCES products(id),
    value       TEXT NOT NULL,          -- зашифрованный ключ
    status      VARCHAR(20) DEFAULT 'available',
    issued_to   INTEGER,
    issued_at   TIMESTAMP
);

-- Заказы
CREATE TABLE orders (
    id          SERIAL PRIMARY KEY,
    user_id     INTEGER NOT NULL,
    product_id  INTEGER REFERENCES products(id),
    amount      DECIMAL(10,2) NOT NULL,
    status      VARCHAR(20) DEFAULT 'pending',
    payment_id  VARCHAR(255),
    created_at  TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP
);

-- Лог выдачи
CREATE TABLE delivery_log (
    id          SERIAL PRIMARY KEY,
    order_id    INTEGER REFERENCES orders(id),
    item_id     INTEGER REFERENCES stock_items(id),
    delivered_at TIMESTAMP DEFAULT NOW(),
    method      VARCHAR(50),            -- email, telegram, api
    status      VARCHAR(20)
);
```

### Логика выдачи (псевдокод)

```python
async def process_payment(payment_data):
    # 1. Проверить подпись webhook
    if not verify_signature(payment_data):
        raise SecurityError("Invalid signature")
    
    # 2. Найти заказ
    order = await get_order(payment_data.order_id)
    if order.status != 'pending':
        return  # уже обработан
    
    # 3. Взять товар из базы (с блокировкой)
    item = await get_available_item(order.product_id, lock=True)
    if not item:
        await notify_admin("Stock empty!", order.product_id)
        await refund(order)
        return
    
    # 4. Выдать
    await deliver(order.user_id, item)
    
    # 5. Обновить статусы
    await mark_item_issued(item.id, order.user_id)
    await mark_order_completed(order.id)
    
    # 6. Залогировать
    await log_delivery(order.id, item.id, method='telegram')
    
    # 7. Проверить остатки
    stock = await get_stock_count(order.product_id)
    if stock < 10:
        await notify_admin(f"Low stock: {stock} remaining")
```

## Заключение

Автоматизация продаж цифровых товаров — это не роскошь и не «когда вырасту». Это необходимость с первого дня.

Три пути:

1. **Готовая площадка** — быстро, бесплатно, ограничения по кастомизации. Подходит для старта.
2. **Telegram-бот** — дёшево, гибко, хватает для среднего объёма.
3. **Своя система** — полный контроль, но нужны разработчик и время.

Главные принципы:

```
→ Выдача должна быть мгновенной (секунды, не часы)
→ Оплата должна проверяться автоматически (не по скриншотам)
→ Каждая операция должна логироваться
→ Остатки должны мониториться
→ Ключи должны храниться зашифрованными
→ Покупатель должен быть защищён (гарант-сервис)
```

Автоматизация не заменяет продукт. Плохой товар не станет хорошим от того, что выдаётся за 3 секунды. Но хороший товар перестанет теряться между «я оплатил» и «ты спишь?».
